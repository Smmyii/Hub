Three Approaches

Pick the architecture that fits. My recommendation is highlighted.

A

Native Zellij Plugin ★ Recommended

A Rust/WASM plugin that lives inside Zellij as a sidebar pane. Full access to the Zellij API — manages tabs, panes, layout, keys. Workspace state persisted in plugin KV store + config files.

Pros

  • Feels like cmux — sidebar is native to Zellij
  • Full API: intercept keys, manage panes, track layout
  • One tool, no external dependencies
  • Claude Code writes the Rust

Cons

  • Rust/WASM compile cycle is slow to debug
  • Zellij plugin API has some rough edges
B

Shell Scripts + Zellij Layouts

Workspaces defined as TOML/YAML config files. Shell scripts create and restore Zellij sessions from those definitions. A fuzzel/rofi picker to switch between them.

Pros

  • No Rust, no WASM — pure shell/fish
  • Very simple to maintain and modify
  • Fast iteration

Cons

  • No sidebar — workspace switching via launcher popup
  • Less integrated, more glued together
  • No live state (can't see what's running per project)
C

External TUI + Zellij Sessions

A standalone TUI app (Go or Python) that manages workspace definitions and spawns Zellij sessions. Floats as a Hyprland window or a fixed Zellij pane.

Pros

  • Friendlier language (Go/Python)
  • More flexible — not limited to Zellij's plugin API
  • Could run without Zellij if needed

Cons

  • More moving parts
  • Less native feel — two separate tools
  • Harder to intercept Zellij keys and state
